## 上下文

- 现状：FastAPI + SQLite，`transactions` 无用户外键，前端单页直接记账。
- 目标：同进程内提供用户认证、角色、管理端 API 与页面；最小依赖。

## 目标 / 非目标

**目标**

- 口令存储使用单向哈希（如 bcrypt 或 Argon2），禁止明文落库。
- 会话采用服务端可校验机制：`HttpOnly` Cookie + 服务端会话表，或带签名的服务端会话 Cookie；须能**登出**并使会话失效。
- `transactions` 表增加 `user_id`（NOT NULL，外键关联 `users`），所有读写限定当前用户；汇总 SQL 带 `user_id` 条件。
- 管理员角色仅用于访问管理端 API 与页面；普通用户不得调用用户管理写接口。

**非目标**

- 密码找回邮件、复杂 RBAC、跨域多子域 SSO。

## 决策

| 决策 | 说明 |
|------|------|
| 用户表 | `users`：`id`，`username`（唯一，区分大小写策略实施时统一为小写存储），`password_hash`，`role`（如 `admin`/`user`），`is_active`（布尔或 0/1），`created_at`，可选 `updated_at`。 |
| 首轮管理员 | 推荐：**无用户时**根据环境变量（如 `LEDGER_BOOTSTRAP_ADMIN_USER` / `LEDGER_BOOTSTRAP_ADMIN_PASSWORD`）创建唯一管理员；若未配置且无用户，服务启动记录明确错误日志或拒绝启动（避免无管理入口），具体须在实施任务中选定其一并文档化。 |
| 管理端形态 | 同应用内 `/admin` 静态页 + `/api/admin/*` JSON；管理员亦需会话，且 `role=admin`。 |
| 普通用户端 | 登录成功后可沿用现有 `/` 记账 UI，或 `/app`；未登录访问记账页时展示登录表单或重定向。 |
| 历史数据 | 升级脚本：为已有 `transactions` 行补充 `user_id`；可默认绑定到首个 `users` 行或由运维手工 SQL——在 `tasks.md` 写清单项。 |

## 风险 / 缓解

| 风险 | 缓解 |
|------|------|
| 会话固定 / 窃取 | `HttpOnly`+`Secure`（HTTPS 下）+`SameSite`；登出清除 Cookie。 |
| 弱口令 | 实施可选最小长度校验；文档提示强口令。 |
| 并发写 SQLite | 保持单进程建议；与现有一致。 |

## 待决问题（实施前可收敛）

- 用户名规则：最小长度与允许字符集在实现时与校验错误文案一并固定。
