## 修改需求

### 需求：浏览器收支录入与汇总展示

系统必须在浏览器中提供界面，使**已登录的当前用户**能够录入**消费**与**收入**交易（每笔含金额），并展示该用户维度下的**累计消费**、**累计收入**以及**净值**（累计收入减去累计消费）。录入成功后，列表与汇总必须与持久化数据一致，且**不得包含任何其他用户的交易**。其他未登录用户不得通过同一套界面或 API 获得该用户的汇总或明细。

#### 场景：录入消费后汇总更新

- **当** 当前登录用户提交一笔类型为消费、金额为正数的交易
- **那么** 该交易出现在该用户的交易列表中，且该用户的累计消费与净值相应更新（净值减少），不影响其他用户的汇总

#### 场景：录入收入后汇总更新

- **当** 当前登录用户提交一笔类型为收入、金额为正数的交易
- **那么** 该交易出现在该用户的交易列表中，且该用户的累计收入与净值相应增加，不影响其他用户的汇总

#### 场景：刷新后数据一致

- **当** 该用户刷新浏览器页面或重新打开同一应用地址且会话仍有效
- **那么** 该用户已录入的交易仍存在，且汇总与单笔明细仍仅反映该用户自身数据

#### 场景：用户间数据不可见

- **当** 用户 A 已登录并查看列表或汇总
- **那么** 响应中不得包含用户 B 的任何交易金额、备注或标识性明细

### 需求：单笔交易补充信息

系统必须为**当前登录用户**名下的每一笔交易提供至少一个可选或可编辑的**补充信息**字段（例如备注/说明），用户可在录入时填写，且必须在界面中能够查看并在后续编辑该信息；保存后再次打开应用且仍以同一用户身份会话访问时必须仍可读取。用户不得编辑或查看归属于其他用户交易的补充信息。

#### 场景：录入时填写补充信息

- **当** 当前登录用户在录入交易时填写补充信息并提交
- **那么** 该用户的列表中展示该笔交易的补充信息内容

#### 场景：事后编辑补充信息

- **当** 当前登录用户选择**本人**已有交易并修改其补充信息后保存
- **那么** 该用户的列表与持久化存储中该交易的补充信息更新为新内容

### 需求：SQLite 持久化

系统必须使用 **SQLite** 作为交易与用户相关数据的持久化存储；交易记录必须通过**用户标识**与所属用户关联存储。除浏览器缓存外，主数据源必须为 SQLite 数据库文件。应用异常退出后，已提交的数据不得丢失。禁止将未落库到 SQLite 的明细作为跨重启后的唯一权威数据。

#### 场景：数据落盘可恢复

- **当** 某用户已成功提交至少一笔交易
- **那么** 在服务进程重启后，该用户仍可在登录后从 SQLite 加载并展示**本人**交易

#### 场景：禁止仅依赖浏览器作为主存储

- **当** 用户清空站点本地存储
- **那么** 收支明细与汇总的权威来源仍须为 SQLite（通过服务端访问）；客户端本地缓存不得替代数据库中的用户隔离数据

### 需求：简洁现代的浏览器界面

系统必须提供**简洁、现代**的页面布局与视觉层次（例如清晰的排版、统一的配色与卡片式分区），交互路径短。普通用户的记账视图须在**已成功登录**后提供；允许单独提供登录表单页或在首屏内嵌登录，但不得在未满足认证要求的同一视图内将匿名用户与已登录用户的记账数据混用。允许适度视觉强调（如渐变背景、轻量过渡）。管理端界面可与用户端区分入口，但必须保持一致的可用性与清晰度。

#### 场景：登录后完成录入与查看汇总

- **当** 普通用户完成登录并进入记账视图
- **那么** 在同一视图或一步可见范围内能够完成录入表单并看到**本人**当前汇总，无需多步无关向导

#### 场景：管理端可达且受控

- **当** 管理员访问管理端入口
- **那么** 管理员可完成用户维护相关操作；非管理员访问该入口时不得获得写权限
