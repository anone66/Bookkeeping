## 新增需求

### 需求：浏览器收支录入与汇总展示

系统必须在浏览器中提供界面，使用户能够录入**消费**与**收入**交易（每笔含金额），并展示**累计消费**、**累计收入**以及**净值**（累计收入减去累计消费）。录入成功后，列表与汇总必须与持久化数据一致。

#### 场景：录入消费后汇总更新

- **当** 用户提交一笔类型为消费、金额为正数的交易
- **那么** 该交易出现在交易列表中，且累计消费与净值相应增加（净值减少）

#### 场景：录入收入后汇总更新

- **当** 用户提交一笔类型为收入、金额为正数的交易
- **那么** 该交易出现在交易列表中，且累计收入与净值相应增加

#### 场景：刷新后数据一致

- **当** 用户刷新浏览器页面或重新打开同一应用地址
- **那么** 已录入的交易仍存在，且汇总与单笔明细一致

### 需求：单笔交易补充信息

系统必须为每一笔交易提供至少一个可选或可编辑的**补充信息**字段（例如备注/说明），用户可在录入时填写，且必须在界面中能够查看并在后续编辑该信息；保存后再次打开应用时必须仍可读取。

#### 场景：录入时填写补充信息

- **当** 用户在录入交易时填写补充信息并提交
- **那么** 列表中展示该笔交易的补充信息内容

#### 场景：事后编辑补充信息

- **当** 用户选择已有交易并修改其补充信息后保存
- **那么** 列表与持久化存储中该交易的补充信息更新为新内容

### 需求：SQLite 持久化

系统必须使用 **SQLite** 作为交易数据的持久化存储；除浏览器缓存外，主数据源必须为 SQLite 数据库文件。应用异常退出后，已提交的数据不得丢失。

#### 场景：数据落盘可恢复

- **当** 用户已成功提交至少一笔交易
- **那么** 在服务进程重启后，这些交易仍可从 SQLite 加载并展示

#### 场景：禁止仅依赖浏览器作为主存储

- **当** 用户清空站点本地存储或未使用服务端数据库
- **那么** 实现不得将未同步到 SQLite 的明细当作唯一权威数据；权威数据必须来自 SQLite（通过服务端访问）

### 需求：简洁现代的浏览器界面

系统必须提供**简洁、现代**的页面布局与视觉层次（例如清晰的排版、统一的配色与卡片式分区），交互路径短，不得在首屏强制多步向导；允许适度视觉强调（如渐变背景、轻量过渡）以符合「炫酷简单」的定位，但不得将复杂动画列为功能前置条件。

#### 场景：首屏完成录入与查看汇总

- **当** 用户打开应用首页
- **那么** 在同一视图中或一步可见范围内能够完成录入表单并看到当前汇总，无需额外登录或多页面跳转
