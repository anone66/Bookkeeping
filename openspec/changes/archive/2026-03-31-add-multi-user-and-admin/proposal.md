# 变更：多用户登录与管理员用户列表

## 为什么

当前实现为单机单用户记账，所有交易无归属。需要**普通用户登录后仅能查看与修改本人的收支**，并由**管理端**维护用户账号列表，以支持多人共用同一部署实例。

## 变更内容

- 增加**用户账号**模型：至少包含唯一登录名、凭据（服务端安全存储）、角色（区分管理员与普通用户）及启用状态；数据存于现有 SQLite 方案中扩展表结构。
- 增加**用户登录与登出**：通过浏览器可建立已认证会话；未认证访问受保护的收支 API 与记账页时须被拒绝或引导登录。
- **管理端**：管理员在独立界面中查看用户列表、创建用户、以及至少支持禁用/删除用户中一种账号级处置（具体以实现阶段 `design.md` 为准，须写入规范场景）。
- **数据隔离**：交易（及汇总）必须与**当前登录用户**绑定；用户 A 的列表与汇总不得包含用户 B 的数据；管理员在规范范围内若未要求「代查他人账本」，则管理端以用户维护为主。
- **与既有变更关系**：本变更对 `add-personal-ledger-web` 中「无登录的记账首页」语义构成**行为升级**；实施后普通用户记账流程以「登录后」为准（见 `personal-ledger` 规范增量）。

## 影响

- **受影响规范**：新增 `admin-users`、`user-auth`；修改 `personal-ledger` 单用户假设（见增量 `spec.md`）。
- **受影响代码**：`app/main.py`、数据库初始化与 `transactions` 表结构、静态前端（登录页 + 管理页 + 现有记账页接会话）；可能新增密码哈希与会话中间件。
- **数据迁移**：已有 `data/ledger.db` 中含历史交易时，必须在实施任务中定义迁移策略（如增加 `user_id` 并归属默认账号或一次性脚本）。

## 不在本次范围

- 多因素认证、OAuth 第三方登录、审计日志、管理员查看任意用户流水（除非后续单独提案）。
