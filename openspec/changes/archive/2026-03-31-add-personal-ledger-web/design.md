## 上下文

- 用户在本机运行程序，通过浏览器访问；数据仅存本地 SQLite。
- 需求强调「炫酷简单」：视觉现代但实现保持最小依赖，避免重型前端框架 unless 实施阶段证明必要。

## 目标 / 非目标

**目标**

- 单笔交易区分类型（消费 / 收入）、金额必填、时间戳可查。
- 汇总实时或刷新后一致（与数据库聚合一致）。
- 补充信息可持久保存，支持查看列表与编辑。
- 单一进程即可启动（例如一个命令启动 API + 静态资源服务）。

**非目标**

- 认证与多租户；审计合规级要求。
- 货币多币种、汇率；预算与图表大盘（可后续提案）。

## 决策

| 决策 | 说明 |
|------|------|
| 存储模型 | 表 `transactions`：`id`，`type`（`expense`/`income`），`amount`（数值，与展示精度一致），`note` 或 `details`（文本，可空），`created_at`，必要时 `updated_at`。金额以「主货币最小单位」或「固定小数位」之一在实现时统一，并在 tasks 中验证。 |
| 架构形状 | **B/S 最小全栈**：HTTP API（REST 或等价 JSON）+ 浏览器静态页调用 API；SQLite 仅由服务端进程打开，避免浏览器直连 SQL。 |
| 技术栈（建议） | 实施后选一种即可：`Python`（FastAPI/Flask）+ `sqlite3`；或 `Node`（Express/Fastify）+ `better-sqlite3`/`sql.js` 仅服务端。前端可用单页 HTML + CSS（现代变量、backdrop-filter 等）+ 少量原生 JS，或极轻量打包。最终以 `tasks.md` 中选定栈为准。 |
| 「炫酷简单」 | 暗色或渐变背景、卡片布局、清晰字体与间距、按钮/列表微过渡；禁止依赖大量动画库作为必选。 |

## 考虑的替代方案

- **纯静态页 + sql.js**：浏览器内 SQLite，部署简单但数据在浏览器存储，备份与一致性弱于服务端单文件库 → 不采用作为主方案。
- **Electron 桌面**：功能过剩，违背「浏览器页面」表述 → 非首选。

## 风险 / 权衡

| 风险 | 缓解 |
|------|------|
| SQLite 文件路径冲突或多进程写锁 | 单进程服务；文档说明勿多实例同库。 |
| 金额浮点误差 | 使用整数分/固定 DECIMAL 策略并在任务中写清。 |

## 迁移计划

- 首版无迁移；日后表结构变更可通过版本化迁移脚本（实施阶段补充）。
