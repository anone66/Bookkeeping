## 需求
### 需求：浏览器收支录入与汇总展示

系统必须在浏览器中提供界面，使**已登录的当前用户**能够录入**消费**与**收入**交易（每笔含金额），并展示该用户维度下的**累计消费**、**累计收入**以及**净值**（累计收入减去累计消费）。录入成功后，列表与汇总必须与持久化数据一致，且**不得包含任何其他用户的交易**。**交易列表接口必须支持分页，接受 `page` 和 `page_size` 参数，返回包含分页元数据的结构化对象而非纯数组。** 其他未登录用户不得通过同一套界面或 API 获得该用户的汇总或明细。

#### 场景：录入消费后汇总更新

- **当** 当前登录用户提交一笔类型为消费、金额为正数的交易
- **那么** 该交易出现在该用户的交易列表中，且该用户的累计消费与净值相应更新（净值减少），不影响其他用户的汇总

#### 场景：录入收入后汇总更新

- **当** 当前登录用户提交一笔类型为收入、金额为正数的交易
- **那么** 该交易出现在该用户的交易列表中，且该用户的累计收入与净值相应增加，不影响其他用户的汇总

#### 场景：分页加载交易列表

- **当** 当前登录用户请求交易列表并指定页码与每页条数
- **那么** 返回结构包含当前页交易数组、总记录数、当前页码和每页条数

#### 场景：刷新后数据一致

- **当** 该用户刷新浏览器页面或重新打开同一应用地址且会话仍有效
- **那么** 该用户已录入的交易仍存在，且汇总与单笔明细仍仅反映该用户自身数据

#### 场景：用户间数据不可见

- **当** 用户 A 已登录并查看列表或汇总
- **那么** 响应中不得包含用户 B 的任何交易金额、备注或标识性明细

### 需求：单笔交易补充信息

系统必须为**当前登录用户**名下的每一笔交易提供至少一个可选或可编辑的**补充信息**字段（例如备注/说明），用户可在录入时填写，且必须在界面中能够查看并在后续编辑该信息；保存后再次打开应用且仍以同一用户身份会话访问时必须仍可读取。用户不得编辑或查看归属于其他用户交易的补充信息。**编辑交易时还必须支持修改交易类型（消费/收入）和交易发生日期。**

#### 场景：录入时填写补充信息

- **当** 当前登录用户在录入交易时填写补充信息并提交
- **那么** 该用户的列表中展示该笔交易的补充信息内容

#### 场景：事后编辑补充信息

- **当** 当前登录用户选择**本人**已有交易并修改其补充信息后保存
- **那么** 该用户的列表与持久化存储中该交易的补充信息更新为新内容

#### 场景：事后修改交易类型与日期

- **当** 当前登录用户选择**本人**已有交易并修改其类型或交易发生日期后保存
- **那么** 该交易的类型或日期在列表与持久化存储中更新为新值，汇总随之重新计算

### 需求：SQLite 持久化

系统必须使用 **SQLite** 作为交易与用户相关数据的持久化存储；交易记录必须通过**用户标识**与所属用户关联存储。除浏览器缓存外，主数据源必须为 SQLite 数据库文件。应用异常退出后，已提交的数据不得丢失。禁止将未落库到 SQLite 的明细作为跨重启后的唯一权威数据。

#### 场景：数据落盘可恢复

- **当** 某用户已成功提交至少一笔交易
- **那么** 在服务进程重启后，该用户仍可在登录后从 SQLite 加载并展示**本人**交易

#### 场景：禁止仅依赖浏览器作为主存储

- **当** 用户清空站点本地存储
- **那么** 收支明细与汇总的权威来源仍须为 SQLite（通过服务端访问）；客户端本地缓存不得替代数据库中的用户隔离数据

### 需求：简洁现代的浏览器界面

系统必须提供**简洁、现代**的页面布局与视觉层次（例如清晰的排版、统一的配色与卡片式分区），交互路径短。界面必须支持深浅色（白天/黑夜）模式切换，并在视觉上结合高级光效等炫酷特效。普通用户的记账视图须在**已成功登录**后提供；允许单独提供登录表单页或在首屏内嵌登录，但不得在未满足认证要求的同一视图内将匿名用户与已登录用户的记账数据混用。管理端界面可与用户端区分入口，但必须保持一致的可用性与清晰度。

#### 场景：登录后完成录入与查看汇总

- **当** 普通用户完成登录并进入记账视图
- **那么** 在同一视图或一步可见范围内能够完成录入表单并看到**本人**当前汇总，无需多步无关向导

#### 场景：管理端可达且受控

- **当** 管理员访问管理端入口
- **那么** 管理员可完成用户维护相关操作；非管理员访问该入口时不得获得写权限

#### 场景：主题切换与视觉特效

- **当** 用户点击主题切换按钮或根据系统偏好
- **那么** 界面可以平滑切换深色与浅色主题，并在卡片、按钮与背景上展示对应的高级流光与毛玻璃特效

#### 场景：从记账页导入第三方账单

- **当** 已登录用户在记账页使用导入入口、选择平台并提交导出文件
- **那么** 用户可在同一记账视境内触发导入并看到结果反馈（成功摘要或错误），且导入完成后列表与汇总反映新数据

### 需求：支付宝与微信账单导入

系统必须为**已登录的当前用户**提供从**支付宝交易明细 CSV**与**微信支付账单流水 XLSX**批量导入交易的能力；导入的每一行经校验后必须写入 SQLite，且**仅归属于该用户**，汇总与列表行为与手工录入一致。除金额、收支类型与交易日期外，系统必须将导出中与对账相关的**结构化字段**落入数据库**独立可空列**（手工录入的记录这些列为空）：支付宝侧包括交易分类、交易对方、商品说明、收/付款方式、商家订单号、导出文件中的备注列等；微信侧包括交易类型、交易对方、商品、支付方式、商户单号、导出备注列等；与同表已有的用户可编辑**补充信息**字段（`note`）区分——`bill_export_note` 仅承载导出备注原文或可解析片段，用户仍可在 `note` 中自行补充说明。列表或详情界面必须允许用户查看这些导入字段，以便与原始账单核对。导入必须通过**记账页**上的统一入口触发：用户点击导入后**下拉选择平台**（支付宝或微信），再选择本地导出文件并提交。服务端必须校验所选平台与实际上传文件格式一致（例如扩展名与表头特征），不一致时必须拒绝请求并返回明确错误。解析必须兼容本项目 `账单/` 目录下提供的官方导出样例结构：支付宝 CSV 在说明性前置行之后出现以「交易时间」为首列的明细表头；微信 XLSX 在说明性前置行之后出现含「交易时间」「收/支」「金额(元)」等的表头行，且「交易时间」列可能以 **Excel 日期序列**形式存储。业务映射规则：**「支出」→ 消费、「收入」→ 收入**；金额必须为正并成功解析为系统使用的货币最小单位；**必须跳过**支付宝「不计收支」与微信「中性交易」对应行；仅导入状态表明**交易已成功完成**的明细（例如样例中的「交易成功」「支付成功」「已转账」等，以实现与固定样例对齐为准）。**幂等**：对同一用户、同一导入来源标识、同一外部交易单号（支付宝「交易订单号」、微信「交易单号」）不得重复插入第二条记录；重复提交同一文件或交叉重复时必须稳定地计入已存在或跳过，而不 inflate 汇总。

#### 场景：支付宝 CSV 导入成功

- **当** 已登录用户选择支付宝平台并上传与样例一致的有效明细 CSV
- **那么** 应写入若干条消费/收入记录（不计收支行不落库），并可在该用户列表与汇总中看到相应更新，且不包含其他用户数据

#### 场景：微信 XLSX 导入成功

- **当** 已登录用户选择微信平台并上传与样例一致的有效流水 XLSX
- **那么** 应写入若干条消费/收入记录（中性交易不落库），交易日期与金额与导出内容一致（时区按导出说明处理），并正确反映在该用户列表与汇总中

#### 场景：平台与文件不匹配

- **当** 用户选择支付宝却上传 XLSX，或选择微信却上传 CSV，或文件表头与所选平台不符
- **那么** 服务端必须拒绝导入且不写入半成批数据，并返回可理解的错误说明

#### 场景：重复外部单号不重复入账

- **当** 同一用户再次导入包含已入库外部单号的文件
- **那么** 不得为相同单号新增重复交易，汇总金额不得被重复计算

#### 场景：刷新后导入数据仍在

- **当** 导入成功后用户刷新页面或重启服务后重新登录
- **那么** 已导入交易仍可从 SQLite 加载并仅对该用户可见

#### 场景：导入字段可查阅与补充信息并存

- **当** 用户已导入一笔交易并在记账列表或详情中查看该笔
- **那么** 应能区分展示账单来源字段（分类/类型、对方、商品摘要、支付方式、商户单号、导出备注等）与用户 `note` 中的补充说明；手工录入的交易不展示空的账单列或应折叠为空状态

### 需求：交易发生时间可选且默认当天

系统必须允许已登录用户在录入交易时指定“交易发生时间”；当用户未主动选择时，系统必须自动使用当天日期作为默认值并参与后续查询与统计。

#### 场景：未选择交易时间时自动默认当天

- **当** 当前登录用户提交一笔交易且未修改交易时间字段
- **那么** 该交易的交易发生时间应保存为提交当日日期，并可在记录详情中查看

#### 场景：手动选择历史日期后按所选日期入账

- **当** 当前登录用户在录入时手动选择某一历史日期并提交交易
- **那么** 该交易必须按该历史日期参与年份月份筛选、分组与汇总计算

### 需求：交易记录按年份月份分组与总体对照汇总

系统必须支持按年份、月份对当前登录用户的交易记录进行筛选和分组展示，并在展示分期汇总的同时保留全量总体汇总以便对照；任一展示或统计结果均不得包含其他用户交易。

#### 场景：按年份查看分组与汇总

- **当** 当前登录用户选择某一年份查看交易记录
- **那么** 系统应展示该年范围内按月份分组的交易列表，并返回该年消费、收入与净值汇总

#### 场景：按年月查看当月分组与汇总

- **当** 当前登录用户选择某一年份与月份查看交易记录
- **那么** 系统应仅展示该月交易，并返回该月消费、收入与净值汇总

#### 场景：分期汇总与总体汇总并行展示

- **当** 当前登录用户已选择任意年份或月份筛选条件
- **那么** 页面中应同时可见筛选范围汇总与总体汇总，且两者统计口径分别对应筛选范围与全量范围

### 需求：交易数据导出

系统必须为**已登录的当前用户**提供将交易记录导出为 CSV 文件的能力。导出必须支持与列表查询相同的筛选参数（日期范围、关键词等），仅导出**当前用户**的数据。导出文件须包含交易类型、金额、日期、备注及导入来源字段（如有）。

#### 场景：导出当前筛选范围的交易

- **当** 已登录用户点击导出并选择当前筛选范围
- **那么** 浏览器下载一个 CSV 文件，内容仅包含该用户在该筛选范围内的交易数据

#### 场景：无数据时导出空文件

- **当** 已登录用户在当前筛选范围内无任何交易记录时尝试导出
- **那么** 系统返回仅含表头的空 CSV 文件，不报错

### 需求：交易关键词搜索

系统必须支持已登录用户按关键词搜索交易记录。搜索范围至少覆盖**备注（note）**、**交易对方（bill_counterparty）**和**商品说明（bill_product）**字段，采用模糊匹配。搜索结果仅包含**当前用户**的数据，可与日期范围筛选组合使用。

#### 场景：按关键词搜索交易

- **当** 已登录用户输入关键词并触发搜索
- **那么** 列表仅展示备注、交易对方或商品说明中包含该关键词的交易记录

#### 场景：无匹配结果

- **当** 已登录用户输入的关键词未匹配任何交易
- **那么** 列表为空并展示相应提示文案

### 需求：用户自助修改密码

系统必须允许已登录的普通用户与管理员自助修改自身口令，无需其他管理员介入。修改口令时必须验证旧口令正确性，验证通过后方可设置新口令。新口令须满足与注册时相同的密码策略。

#### 场景：输入正确旧密码后成功修改

- **当** 已登录用户提供正确的旧口令和满足策略的新口令
- **那么** 口令更新成功，后续须使用新口令登录

#### 场景：旧密码错误时拒绝修改

- **当** 已登录用户提供的旧口令不正确
- **那么** 系统拒绝修改并返回错误提示
