## 修改需求

### 需求：用户登录与登出

系统必须允许用户通过**登录名与口令**建立已认证会话，并在用户主动登出或会话失效后结束访问受保护资源的能力。口令不得以明文形式持久存储于数据库中。登录失败时不得区分「用户不存在」与「口令错误」之外的敏感细节（可实现为统一错误提示）。**口令字符集必须至少支持 ASCII 字母、数字及常用特殊字符（如 `!@#$%^&*()-_+=~` 等），最短长度须为 6 位。** **登录接口必须实施速率限制，在短时间窗口内对同一来源的连续失败尝试达到阈值后拒绝后续请求并返回 429 状态码，以防止暴力破解。**

#### 场景：登录成功获得会话

- **当** 用户提供正确的登录名与口令且账号处于可登录状态
- **那么** 系统建立会话，随后的受保护请求在该会话有效期内被识别为该用户

#### 场景：登录失败

- **当** 用户提供错误或不可用的凭据
- **那么** 系统不建立会话，并返回与成功路径明确区分的错误结果

#### 场景：登出

- **当** 已登录用户执行登出操作
- **那么** 后续请求不再视为已认证（除非再次登录）

#### 场景：暴力破解防护

- **当** 同一来源在短时间窗口内连续登录失败超过阈值
- **那么** 后续登录请求被拒绝并返回 429 状态码，冷却期结束后恢复正常

#### 场景：含特殊字符的口令可正常使用

- **当** 用户使用包含 `!@#$%^&*()-_+=~` 等特殊字符的口令注册或登录
- **那么** 系统正确处理该口令的存储与验证，不因字符集限制而拒绝

## 新增需求

### 需求：CSRF 防护

系统必须为所有状态变更接口（POST / PATCH / DELETE）实施 CSRF 防护机制。Cookie 会话认证的请求必须携带有效的 CSRF 令牌，否则服务端必须拒绝请求并返回 403 错误。CSRF 令牌必须在每个会话中唯一生成，前端在发起状态变更请求时必须自动附带该令牌。

#### 场景：缺少 CSRF 令牌时拒绝请求

- **当** 客户端发送 POST/PATCH/DELETE 请求但未携带有效的 CSRF 令牌
- **那么** 服务端拒绝该请求并返回 403 错误

#### 场景：携带有效 CSRF 令牌时正常处理

- **当** 客户端在请求中正确携带当前会话的 CSRF 令牌
- **那么** 服务端正常处理该请求

## 新增需求

### 需求：会话过期清理

系统必须实现过期会话记录的清理策略，禁止 `sessions` 表无限膨胀。可采用启动时批量清理、定时清理或请求时懒清理中的一种或多种组合方式。

#### 场景：过期会话被清理

- **当** 系统启动或在运行过程中触发清理逻辑
- **那么** `sessions` 表中 `expires_at` 早于当前时间的记录被删除，不影响有效会话

## 新增需求

### 需求：安全响应头与 Cookie 加固

系统必须在 HTTP 响应中设置 `Content-Security-Policy`、`X-Content-Type-Options: nosniff`、`X-Frame-Options: DENY` 等安全头。当检测到 HTTPS 部署环境时，会话 Cookie 必须自动附加 `Secure` 标志，防止通过明文连接泄露。

#### 场景：安全响应头存在

- **当** 客户端向任意接口发起请求
- **那么** 响应中包含 `Content-Security-Policy`、`X-Content-Type-Options` 和 `X-Frame-Options` 头

#### 场景：HTTPS 环境下 Cookie 标记 Secure

- **当** 服务运行在 HTTPS 代理后方且请求标识为安全连接
- **那么** 设置会话 Cookie 时包含 `Secure` 标志
