## 修改需求

### 需求：浏览器收支录入与汇总展示

系统必须在浏览器中提供界面，使**已登录的当前用户**能够录入**消费**与**收入**交易（每笔含金额），并展示该用户维度下的**累计消费**、**累计收入**以及**净值**（累计收入减去累计消费）。录入成功后，列表与汇总必须与持久化数据一致，且**不得包含任何其他用户的交易**。其他未登录用户不得通过同一套界面或 API 获得该用户的汇总或明细。**汇总接口的实现不得对同一统计范围执行重复的 SQL 查询；无筛选时总体汇总与范围汇总共用同一次查询结果即可。**

#### 场景：录入消费后汇总更新

- **当** 当前登录用户提交一笔类型为消费、金额为正数的交易
- **那么** 该交易出现在该用户的交易列表中，且该用户的累计消费与净值相应更新（净值减少），不影响其他用户的汇总

#### 场景：录入收入后汇总更新

- **当** 当前登录用户提交一笔类型为收入、金额为正数的交易
- **那么** 该交易出现在该用户的交易列表中，且该用户的累计收入与净值相应增加，不影响其他用户的汇总

#### 场景：刷新后数据一致

- **当** 该用户刷新浏览器页面或重新打开同一应用地址且会话仍有效
- **那么** 该用户已录入的交易仍存在，且汇总与单笔明细仍仅反映该用户自身数据

#### 场景：用户间数据不可见

- **当** 用户 A 已登录并查看列表或汇总
- **那么** 响应中不得包含用户 B 的任何交易金额、备注或标识性明细

## 修改需求

### 需求：支付宝与微信账单导入

系统必须为**已登录的当前用户**提供从**支付宝交易明细 CSV**与**微信支付账单流水 XLSX**批量导入交易的能力；导入的每一行经校验后必须写入 SQLite，且**仅归属于该用户**，汇总与列表行为与手工录入一致。除金额、收支类型与交易日期外，系统必须将导出中与对账相关的**结构化字段**落入数据库**独立可空列**。**账单解析模块的内部辅助函数不得在循环体内反复定义；须在循环外定义一次，以 row 为参数或闭包引用迭代变量。**

#### 场景：支付宝 CSV 导入成功

- **当** 已登录用户选择支付宝平台并上传与样例一致的有效明细 CSV
- **那么** 应写入若干条消费/收入记录（不计收支行不落库），并可在该用户列表与汇总中看到相应更新，且不包含其他用户数据

#### 场景：重复外部单号不重复入账

- **当** 同一用户再次导入包含已入库外部单号的文件
- **那么** 不得为相同单号新增重复交易，汇总金额不得被重复计算
